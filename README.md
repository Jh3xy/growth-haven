> **Personal project.** GrowthHaven is a self-directed full-stack development project built to explore product design, frontend architecture, and backend integration. It is not a registered business, not a financial product, and is not intended to collect, hold, or process real money. Any reward figures shown (₦ amounts) are in-platform credits used purely for demonstration purposes within a closed system.

Live: [growthhaven.app](https://growthhaven.app)

---

## What it is

GrowthHaven is a creator-focused web platform where users earn in-platform rewards through content, community activity, and engagement. It was built as a personal project to demonstrate a complete product — from public marketing site through authenticated dashboard, role-based portals, and a multi-feature member experience.

The platform is structured around six environments:

| Environment | What it does |
|---|---|
| **Creator Hub** | Personal dashboard — wallet balance, referral tracking, transaction history, profile management |
| **Writer's Circle** | Post creation and community feed. Members earn rewards per like on their posts and for liking others |
| **Stream & Earn** | Embedded music player sourcing content from YouTube, Vevo, MTV Base, and Audiomack. Completed stream sessions earn rewards |
| **Community Rewards** | Tiered referral system. Members earn a percentage cut from activity generated by people they refer |
| **The Arcade** | Five browser-based arcade games (Coin Flip, Dice, Limbo, Plinko, Mines). Points-based, no real-money wagering |
| **Quests** | Daily and weekly challenge system. Members complete tasks to earn incremental rewards |

## Tech stack

### Frontend
- **Build tool:** Vite (multi-page app / MPA configuration)
- **Languages:** HTML, CSS, vanilla JavaScript — no frameworks
- **Icons:** Lucide via CDN
- **Font:** General Sans (self-hosted)
- **CSS architecture:** Modular files (`variables.css`, `normalize.css`, `fonts.css`, `utils.css`, `style.css`, `landing.css`, `animations.css`) — imported through `script.js` via Vite, not via `<link>` tags
- **Design system:** Custom CSS design tokens (light + dark theme), mobile-first layout

### Backend
- **Platform:** [Supabase](https://supabase.com)
- **Auth:** Supabase Auth (email/password, OTP, password recovery)
- **Database:** Postgres via Supabase
- **Deployment:** Vercel (frontend), Supabase cloud (backend)

## Supabase architecture

The entire backend runs on Supabase. Here is what it handles:

### Authentication
Supabase Auth manages the full auth lifecycle — registration, login, OTP verification, and password recovery. A `handle_new_user` Postgres trigger fires on every new `auth.users` insert and automatically populates the `members` table with the user's first name, last name, and a generated referral code. Referral codes follow the format `XX-XXXXX` (initials-prefixed, custom character pool avoiding ambiguous characters, collision loop for uniqueness).

### Database tables (public schema)
- `members` — core user profile table. Linked 1:1 to `auth.users`. Stores referral code, referrer code, display info, and wallet balance
- `promoters` — extended table for Promoter Partner tier users. Stores commission rate and wallet balance
- `posts` — Writer's Circle content. Tracks author, body, image, like count
- `post_likes` — junction table for post likes with rate limiting logic
- `music_catalog` — track metadata for the Stream & Earn player
- `stream_sessions` — records stream activity per user per track for reward calculation
- `quests` / `quest_completions` — quest definitions and per-user completion records
- `arcade_sessions` — game round history per user
- `investment_plans` — growth plan records (create, daily claim, early exit)
- `transactions` — unified ledger for all credit/debit activity across the platform
- `waitlist` — captures pre-registration form submissions (name, email, phone)
- `withdrawals` — withdrawal request records pending admin review

### Row Level Security (RLS)
RLS is enabled across all user-facing tables. Policies are scoped so users can only read and write their own records. Admin-level access is handled separately.

### SECURITY DEFINER functions
Several Postgres functions run with `SECURITY DEFINER` to allow controlled cross-table reads that would otherwise be blocked by RLS:
- `get_my_referrals()` — returns referral list for the current user
- `get_promoter_referrals()` — returns enriched referral data for Promoter Portal users, including last-active timestamps
- `start_stream()` / `complete_stream()` — manage stream session state and credit reward on completion
- `get_stream_user_status()` — returns current stream eligibility and session state for a user

### Triggers
- `handle_new_user` — fires `AFTER INSERT` on `auth.users`. Creates the member profile, generates referral code, and records the referrer relationship if a referral code was supplied at registration

## Auth and routing

- All protected pages check for an active Supabase session on load and redirect to `/src/login/` if none exists
- The Promoter Portal (`/src/affiliate/`) additionally checks the `promoters` table for a valid record and redirects non-promoters to the standard dashboard
- The Admin dashboard is separately gated
- Referral codes in registration URLs are captured from query params and passed to the `handle_new_user` trigger via user metadata

## Running locally

```bash
# Clone the repo
git clone https://github.com/Jh3xy/growth-haven.git
cd growth-haven

# Install dependencies
npm install

# Add environment variables
cp .env.example .env
# Fill in your Supabase project URL and anon key

# Start dev server
npm run dev
```

You will need your own Supabase project with the schema applied. The `/supabase` folder contains the relevant migrations and function definitions.
